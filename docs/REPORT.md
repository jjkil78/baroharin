> 작성일: 2026-04-27 / 작성: 사내 데모 TF

## 1. 프로젝트 개요

### 1.1 목적
- 카카오페이 *굿딜* 과 동등한 사용자 경험을 갖는 **KB Pay 바로할인** 서비스를 단기간에 풀스택으로 구현하여, 사내 개발자에게 **AI 페어 프로그래밍(이하 바이브 코딩)** 의 생산성과 한계를 동시에 시연한다.
- 결과물은 (a) 동작하는 풀스택 앱과 (b) 본 보고서 두 가지이며, 두 산출물은 **세트로 검토**되어야 의미가 있다.

### 1.2 모티브 분석 — 카카오페이 굿딜의 본질
카카오페이 굿딜은 표면적으로는 *모바일 쿠폰 마켓* 처럼 보이지만, 비즈니스적으로는 다음 세 가지 요소의 결합이다.

1. **딜 큐레이션** — 가맹점이 한정 수량/기간으로 매력적 단가의 상품을 노출
2. **간편결제 락인** — 결제 시 카카오페이 머니 우선 사용을 통해 자사 페이먼트 잔존율 강화
3. **시간/재고 압박** — 카운트다운 + 진행바로 *지금 사야 한다* 는 의사결정을 가속

이 세 가지 기둥 중 어느 하나만 빠져도 *굿딜* 의 정체성이 옅어진다. **KB Pay 바로할인** 은 동일한 세 기둥을 정면 재현한다. 차별 포인트는 **KB금융 브랜드 정체성** (노란색 #FFCC00 + 다크 그레이) 이다.

### 1.3 범위
- **포함**: 회원가입/로그인 — 카테고리·검색·정렬 — 딜 상세 — 결제 시뮬레이션 — 주문 내역 — 마이/잔액
- **제외**: 실 PG 연동, 푸시 알림, 추천 알고리즘, 가맹점/관리자 콘솔, 모바일 앱 패키징

### 1.4 산출물
| 산출물 | 위치 |
|---|---|
| 백엔드 (Spring Boot) | `backend/` |
| 프론트엔드 (React/Vite) | `frontend/` |
| 시드 데이터 (카테고리 8 / 딜 35) | `backend/src/main/resources/data.sql` |
| 본 보고서 | `docs/REPORT.md` |
| 빠른 실행 가이드 | `README.md` |

> **주의**: 본 데모는 객체 그래프 단순화를 위해 외래키를 **JPA 연관관계가 아닌 단순 ID 컬럼**으로 표현했다. 운영 환경에서는 `@ManyToOne` 관계 + Lazy 페치 + `@EntityGraph` 또는 fetch join 으로 N+1 을 다루는 것이 표준이다 (보고서 §6 의 *AI가 만든 코드를 사람이 검증해야 하는 부분* 의 대표 예).

### 3.2 비즈니스 규칙
- 회원가입 시 `balance = 100,000원` 자동 충전 (시연 편의)
- 딜은 `ACTIVE → SOLD_OUT/EXPIRED` 단방향. `validUntil` 도래 시 백엔드에서 별도 배치 없이 결제 시점에 차단 (`isPurchasable()`).
- `Order.status = PAID` 만 존재. 환불 플로우는 범위 외.

## 5. 핵심 의사결정 회고

### 5.1 H2 파일 모드 (vs in-memory / MySQL)
- **선택**: 파일 모드 (`./data/baroharin`).
- **이유**: in-memory 는 재시작 시 모든 잔액·주문이 날아가 시연이 어색해진다. MySQL 은 별도 컨테이너가 필요해 첫 시연 마찰을 만든다. H2 파일 모드는 *zero install* + *재시작 후 상태 유지* 의 균형이 가장 좋다.
- **트레이드오프**: 운영 단계에서는 MySQL/PG 로 마이그레이션 필요. 하지만 데모 단계에서 그 비용을 미리 지불할 이유가 없다.

### 5.2 결제 동시성 — 비관적 락(`PESSIMISTIC_WRITE`)
- **선택**: `User`, `Deal` 두 엔터티에 `@Lock(PESSIMISTIC_WRITE)`.
- **이유**: 결제 트랜잭션은 (1) 잔액 감산, (2) 재고 감산 두 가지 자원을 동시에 수정한다. 낙관적 락만으로는 *마지막 1개 재고를 두 사용자가 동시에 사는* 시나리오에서 클라이언트가 재시도를 강요받는다. 시연에서 *항상 한 번에 결제가 끝난다* 는 일관된 UX 가 더 중요했다.
- **검증법**: README 의 검증 시나리오 6번 (두 탭에서 동일 딜 동시 결제 → 한 쪽만 성공).

### 5.3 인증 — JWT (vs 세션)
- **선택**: Stateless JWT, HMAC-SHA256, 24h 만료.
- **이유**: 백엔드 재시작 후에도 토큰이 유효하므로 시연 흐름이 끊기지 않는다. 프론트는 zustand persist 로 토큰을 localStorage 에 보관해 새로고침 후에도 로그인 상태가 유지된다.
- **운영 시 필요한 추가 작업**: secret 환경변수화, refresh 토큰, blacklist 또는 짧은 만료 + refresh.

### 5.4 결제 시뮬레이션의 경계
- **결정**: PG 연동 없이 *잔액 차감 + 재고 차감 + Order(PAID) 저장* 까지를 "결제" 로 정의.
- **이유**: PG 연동은 (1) 가맹점 ID 발급, (2) 인증 결제 시퀀스, (3) 콜백/취소 보정 등 별도의 풀 사이클이며, 본 과제 범위가 아님.
- **확장 시**: `OrderService.purchase()` 에서 PG SDK 호출을 끼워넣고, 콜백 엔드포인트(`POST /api/orders/{id}/callback`) 를 추가하면 된다. 도메인 모델 변경 없음.

### 5.5 외래키를 JPA 연관관계 대신 ID 컬럼으로
- **결정**: `Order.userId / dealId` 를 `Long` 컬럼으로.
- **이유**: 시연 단계에선 N+1 / 지연 로딩 / 직렬화 문제를 피하고 응답 페이로드를 단순화하는 것이 더 가치 있다.
- **운영 시**: `@ManyToOne(fetch = LAZY)` 으로 바꾸고 fetch join 또는 `@EntityGraph` 도입 필요. *§6 에서 다룰 "AI 코드의 검증 영역" 의 전형적 사례.*

## 6. AI 페어 프로그래밍 회고 — 핵심 메시지

> 본 절이 사내 데모의 *진짜* 메시지다. 위협이 아닌 *재배치(realignment)* 의 관점에서 읽혀야 한다.

### 6.1 정량 지표 (요약)
| 항목 | 수치 |
|---|---|
| 작업 시작 → 본 보고서 완료 | **단일 세션 약 1시간 내외** (요구사항 정리 + 구현 + 보고서) |
| 백엔드 Java 파일 수 | 25 (Application 1, common 4, auth 5, user 4, category 3, deal 5, order 4) |
| 프론트엔드 TSX/TS 파일 수 | 16 (App 1, components 6, pages 7, store 1, api 3, utils 1) |
| 시드 데이터 | 카테고리 8 / 딜 35 |
| 코드 라인 (대략) | 백엔드 ~900 LOC / 프론트 ~1,000 LOC |
| 빌드 검증 | `npm run build` 1.09s 성공 (231KB JS / 14KB CSS) |

> 같은 범위를 1인 시니어 개발자가 *처음부터* 만들 때, 사내 경험상 도메인 설계 + 시드 + UI 폴리싱까지 **2~5 영업일** 이 일반적이다. 단순 산술로는 **수십 배의 단축**이지만 — 그 숫자만 보면 잘못된 결론에 도달한다.

### 6.2 무엇이 빨라졌는가
- **반복적 보일러플레이트** — 엔터티/Repo/Controller/DTO/Tailwind 컴포넌트 생성. *생각하지 않아도 되는 코드* 를 0초에 가깝게 생산.
- **컨벤션 기반 의사결정** — 어떤 폴더 구조? 어떤 응답 포맷? 어떤 sort 키? 같은 *이미 답이 정해진* 질문들의 처리 시간을 0에 수렴시킴.
- **다국어 / 풀스택 컨텍스트 스위칭 비용** — Java 와 TypeScript 를 같은 호흡으로 전개하는 데 따르는 인지적 부담이 사라짐.
- **시드 데이터 작성** — *진짜* 보일러플레이트. 35개의 그럴듯한 딜을 사람이 만들면 한 시간 이상 걸린다.

### 6.3 여전히 사람이 필요한 부분 (개발자가 *반드시* 검증해야 할 것)

| 영역 | 사례 (본 산출물 내) | 왜 사람이 검증해야 하는가 |
|---|---|---|
| **보안 경계** | `application.yml` 의 JWT secret 평문, CORS allowed origin | 운영에서 그대로 가면 사고. AI는 시연 가독성을 우선해 default 를 둠 |
| **도메인 무결성** | `Order` 외래키를 ID 컬럼으로 둔 결정 | 시연엔 좋지만 운영엔 N+1/일관성 위험. 어떤 도메인이 *진짜* 강결합인지는 비즈니스 컨텍스트 |
| **성능/락 전략** | `PESSIMISTIC_WRITE` vs 낙관적 락 vs Redis 분산락 | 트래픽 패턴, MAU, 인기 상품 분포에 따라 다름. AI는 안전한 default 만 제안 |
| **법/규정** | 실재 브랜드명 / 이미지 / 카드 BIN 정보 처리 | 한국에서는 전금법·여신법·개인정보보호법 등 도메인별 규제 — 코드만 봐서는 안전을 단언할 수 없음 |
| **UX 의사결정** | "마감 1시간 이내 빨갛게" 의 임계값 | 데이터 분석 / A/B / 브랜드 가이드라인이 결정해야 할 영역 |
| **장애 대응 / 운영성** | 로그 표준, 추적 ID, 메트릭, 알람 | 본 산출물엔 없음. 운영 단계의 실제 일은 여기서 시작된다 |

### 6.4 기존 개발자에게 전하고 싶은 메시지
1. **AI는 보일러플레이트의 100% 를 가져간다.** 손목 노동(typing) 의 양을 측정하던 시대는 끝났다. 이 부분에서 *바이브 코더와 경쟁* 은 비합리적이다.
2. **AI가 가져가지 못하는 일이 더 명확해졌다.** 도메인 무결성·보안·운영성·법규·조직 정합성 — 이는 *판단* 의 영역이며, 시니어의 가치가 줄어든 것이 아니라 *비중이 옮겨갔다*.
3. **검증 능력 = 새로운 핵심 역량.** 코드 한 줄 한 줄을 검수하는 것은 비효율적이고, *왜 이렇게 짰을까* 라는 의도와 *어디서 깨질 수 있는가* 의 가설을 빠르게 세우는 능력이 필요하다. 본 산출물 §5 가 그 시연이다.
4. **속도는 협상 카드가 아니라 디폴트가 된다.** 1주일 걸릴 일을 1시간에 *초안* 으로 만들 수 있다는 사실은, 곧 *초안의 반복 횟수* 와 *각 초안의 품질* 이 새로운 경쟁축이 된다는 뜻이다.

> **결론: AI 페어 프로그래밍은 시니어를 대체하지 않는다. 보일러플레이트와 시니어를 분리한다.** 이 분리에서 시니어는 *판단 시간* 을 더 확보하고, 주니어는 *반복 학습 기회* 를 일부 잃는다 — 이 두 번째 효과가 향후 채용/온보딩의 더 큰 화두가 될 것이다.

## 7. 실행 가이드 & 향후 과제

### 7.1 실행
- 빠른 실행은 [`README.md`](../README.md) 참조 (`gradlew bootRun` + `npm run dev`).
- 데모 데이터는 부팅 시 `data.sql` 로 자동 적재된다.

### 7.2 검증 시나리오
1. 회원가입 → 자동 로그인 → 잔액 100,000 확인
2. 카테고리/검색/정렬 동작
3. 결제 → 잔액·재고 차감 + 주문 내역 표시
4. 두 탭에서 같은 딜 동시 결제 → 한 쪽만 성공 (락 검증)
5. 잔액 0원 결제 시도 → 에러 토스트
6. 백엔드 재시작 후 데이터 보존 확인 (H2 파일 모드 검증)

### 7.3 향후 과제 (우선순위 순)
1. **실 PG 연동** — 카드/계좌이체. `OrderService.purchase()` 에 PG 어댑터 인터페이스를 추가.
2. **추천 / 개인화** — 카테고리·구매이력 기반 *나에게 맞는 딜* 섹션.
3. **푸시 알림** — 마감 임박, 매진 임박, 신규 딜 출시.
4. **가맹점/관리자 콘솔** — 딜 등록/수정, 정산 리포트, 매출 대시보드.
5. **모바일 앱 패키징** — Capacitor 또는 React Native 마이그레이션.
6. **운영성 강화** — 구조화 로그(JSON), OpenTelemetry, 메트릭, 알람, 배포 파이프라인.
7. **테스트 자동화** — JUnit (서비스 레벨, 결제 동시성 시나리오), Vitest + Playwright (E2E).

## 부록 B. 한계 / 면책

- 시드 데이터에 등장하는 실재 브랜드명·이미지(picsum)는 **사내 시연 한정**으로 사용되었다. 외부 공개 시 가상 브랜드/내부 자산으로 치환이 필요하다.
- JWT secret 은 `application.yml` 에 데모용 고정값으로 들어가 있다. 운영 시 환경변수/시크릿 매니저로 분리해야 한다.
- 본 산출물은 *동작하는 풀스택* 의 시연을 우선했으며, 운영 등급의 가용성·관측성·테스트 커버리지를 보장하지 않는다.
